#### 18/01/16 OGC pide la revisión de los candidatos a estándar Geospatial User Feedback Conceptual Model y XML Encoding
OGC ha publicado la documentación de los estándares: Modelo conceptual para comentarios de los usuarios a recursos geoespaciales (OGC Geospatial User Feedback Conceptual Model (GUF Model)) y Codificación XML para comentarios de los usuarios a recursos geoespaciales (OGC Geospatial User Feedback XML encoding) para recabar sugerencias sobre ambos documentos.
El estándar Geospatial User Feedback Conceptual Model especifica la estructura conceptual y el modelo de datos para comentarios de los usuarios a recursos geoespaciales (geospatial user feedback). Los comentarios de los usuarios a recursos geoespaciales son un tipo de metadatos que es producido predominantemente por los consumidores de productos de datos geoespaciales a medida que ganan experiencia con estos productos. Esto complementa los metadatos existentes producidos por el creador, editor, o el conservador de los conjuntos de datos y los detalles que provienen de los flujos de producción implicados en la creación de los datos.
Disponer de este tipo más informal de metadatos tiene como objetivo final recopilar requisitos para datos que los proveedores pueden incorporar como medidas de calidad objetivas para sus productos, de modo que los proveedores sean capaces de cubrir mejor las necesidades reales de los usuarios y potencialmente descubrir nuevos mercados para sus datos.
El modelo GUF específica un modelo que incluye un conjunto de mecanismos amplios para almacenar feedback, incluyendo observaciones en texto libre, puntuaciones, citaciones una publicación, información adicional sobre linaje o acontecimientos significativos que condicionen la interpretación del conjunto de datos.
El modelo GUF reutiliza algunos elementos de la norma de metadatos ISO 19115-1 (la versión actualizada del tema 11 del OGC Abstract Specification), pero no su estructura completa. La selección de estos elementos, se he realizado para optimizar la futura interoperabilidad con la evolución de los modelos en las nuevas normas de metadatos ISO. Se ha realizado un esfuerzo para permitir que las implementaciones del modelo GUF que sean lo más simples posible, de modo que el estándar pueda ser implementado en interfaces de usuario simples diseñadas para distintos niveles de experiencia. El feedback puede ser proporcionado tanto sobre datos como sobre metadatos.
La codificación para una implementación práctica del modelo conceptual se proporciona en un documento a parte que explica como codificar feedback geoespacial en XML. En un futuro, otras codificaciones podrán ser desarrolladas, incluyendo alternativas como el uso de JSON-LD.
Ambos estándares están basados en los trabajos realizados por dos proyectos del 7º Programa Marco (FP7): GeoViQua (FP7 FP7/2007-2013 número de acuerdo 265178) y CHARMe (FP7/2007-2013 número de acuerdo 312641). La información presentada hoy es el resultado de la maduración y el consenso en el seno del Comité Técnico OGC de los citados trabajos iniciales.
La documentación del estándar está disponible en el siguiente enlace www.opengeospatial.org/standards/requests/144. El plazo para enviar comentarios está abierto hasta el próximo 17 de febrero.
07/01/16 Discrete Global Grid Systems (DGGS) Core Standard disponible para revisión
OGC ha publicado la documentación del estándar candidato Discrete Global Grid Systems (DGGS) Core para recabar sugerencias y comentarios. El plazo está abierto hasta el próximo 4 de febrero.
El estándar DGCS define un conjunto de reglas para la definición de arquitecturas altamente eficientes para el almacenamiento de datos espaciales y analíticos. El objetivo del estándar es permitir una integración rápida de datos espaciales sin las dificultades de trabajar con sistemas de coordenadas tradicionales. DGGSs representa la Tierra como secuencias de teselaciones de celdas con coberturas globales y con una resolución espacial progresivamente más fina. Observaciones individuales se pueden asignar a una celda correspondiente a la posición y al tamaño del fenómeno observado. DGGS viene con un conjunto estándar de algoritmos funcionales que permiten un análisis de datos rápido de un gran número de celdas.
Para más información véase http://www.opengeospatial.org/pressroom/pressreleases/2349
21/12/15 OGC pide la revisión del estándar Coverage Implementation Schema (CIS)
OGC ha publicado la documentación del estándar Coverage Implementation Schema (abreviado CIS) para recabar sugerencias y comentarios. El estándar propuesto complementa OGC Abstract Topic 6 Standard (que es idéntico a la norma ISO 19123) con un modelo de cobertura concreto. ISO ha expresado el interés para adoptar CIS 1.1 como ISO 19123-2.
El estándar Coverage Implementation Schema versión 1.1 es sucesor retrocompatible de GML 3.2.1 Application Schema Coverages versión 1.0 (abreviado GMLCOV). Incorporando todos los elementos de GMLCOV 1.0, CIS 1.1 añade nuevas capacidades de modelización, como división de coberturas, interpolación, pares geometría/valor, soporte del estándar SensorML, y nube de puntos. Además, un modelo general para todas las combinaciones de mallas no regulares, que incluye tipos de coberturas GML 3.3 como casos especiales.
OGC Abstract Topic 6 Suite define un modelo de cobertura abstracto. Las coberturas representan colecciones homogéneas de valores localizadas en el espacio/tiempo, como sensores espacio-temporales, imágenes, simulación y datos estadísticos. Ejemplos comunes incluyen series temporales 1-D, imágenes 2D, imágenes de series temporales 3D o modelos espaciales, así como datos sobre climatología y oceanografía en 4D. Generalmente, las coberturas incluyen rejillas regulares e irregulares multi-dimensionales, nubes de puntos y mallas.
Coverage Implementation Schema complementa OGC Abstract Topic 6 con una definición de estructura de cobertura concreta de un subconjunto de modelos de coberturas en OGC Abstract Topic, independientemente del formato de codificación de los datos de las implementaciones. CIS permite un rango amplio de implementaciones interoperables.
Las coberturas pueden ser codificadas en formatos (como GML, JSON, GeoTIFF o CF-netCDF) y pueden ser divididas, por ejemplo por una representación de tiempo intercalado. Las coberturas son independientes de las definiciones de servicio y por lo tanto son accesibles a través de una variedad de tipos de servicios OGC, como el Web Coverage Service (WCS). La estructura de cobertura puede servir un amplio rango de dominios de aplicación de coberturas, lo que contribuye a la armonización y la interoperabilidad entre y a través de estos dominios.
La documentación del estándar está disponible en el siguiente enlace http://www.opengeospatial.org/standards/requests/142 . El plazo está abierto hasta el próximo 16 de enero de 2016.
04/11/15 OGC solicita la participación en un Experimento de Interoperabilidad: GeoPackage (GPKG) Elevation Extension Interoperability Experiment (GPKG-EE IE)
OGC ha anunciado una llamada a la participación en un Experimento de Interoperabilidad sobre Geopackage con datos de elevaciones, GeoPackage (GPKG) Elevation Extension Interoperability Experiment (GPKG-EE IE).
El grupo GeoPackage Standards Working Group (SWG) ha identificado la necesidad de almacenar datos de elevaciones en Geopackage. Con este objetivo, el SWG ha desarrollado una extensión al estándar de codificación para soportar estos datos.
Esta capacidad se utilizará en casos de uso como los siguientes:
Visualización
2D (sombreado, relieve en color, pendiente)
3D ( soporte de cambio de vista y nivel de detalle)
Análisis
Visibilidad
Movilidad entre países
Idoneidad de sitio y planificación (análisis de pendientes en zonas de aterrizaje)
Representaciones de objetos en geometría 3D (terrestre ,aéreo)
asociación del terreno (asociación de imágenes con mapeo de localizaciones)
realidad Aumentada
El grupo propone la validación de la extensión mediante un experimento de interoperabilidad. Este experimento testeará y refinará la extensión. Para ello, el grupo necesitará el apoyo de una comunidad globalmente representativa que proporcione tanto dominio como experiencia técnica a este proyecto. El SWG necesita un núcleo de miembros OGC para apoyar e iniciar este trabajo, los que no son miembros pueden participar como 'observers'.
OGC pide a las organizaciones interesadas indicar su apoyo y compromiso en este Experimento de Interoperabilidad. 25/06/15 OGC anuncia la formación del grupo Hydrologic Features Standards Working Group
Open Geospatial Consortium solicita comentarios del documento fundacional de un nuevo grupo Hydrologic Features HY_Features Standards Working Group (SWG) con el objetivo de progresar el modelo HY_FEATURES al estado de un estándar OGC para una identificación común, estable y la referencia de elementos hidrológicos.
La información hidrológica tiene que ser compartida entre muchas organizaciones. HY_Features conjunto con el estándar WaterML2 proporciona una base de referencia para mejorar la manera en que la información hidrológica es organizada, gestionada y compartida.
Objetos hidrológicos son la unidad de estudio de la información hidrológica, y un medio es requerido para identificar los objetos del mundo real a través de la cadena de procesamiento de datos de observaciones de información hidrográfica. El trabajo iniciado en el Grupo temático Hidrology DWG dirige una serie de estándares y buenas prácticas para gestionar a diferentes niveles de detalle la información, observación y representación de elementos hidrológicos. Ejemplos como el WaterML2.0, modelo de objetos HY_Features, y el trabajo que actualmente se desarrolla de GroundWaterML. Cada estándar está concebido con diferentes aspectos de hidrología. En conjunto estos estándares tienen un entendimiento común del fenómeno hidrológico, que proporciona las bases para relacionar conceptos específicos de la aplicación por referencia a semántica común. Esto permite que sistemas de información y servicios web como ontologías de dominio específico (como los términos usados en los productos de datos existentes) puedan enlazarse usando un modelo de referencia común.
El estándar Hydrology Feature se dividirá en tres partes:
Parte 1: HY_Features conceptual model (OGC 14-111).El modelo normativo es un modelo Unified Modeling Language (UML).
Parte 2: Un esquema de implementación GMLpara transferir información de instancias de objetos HY_Features, basado en la Norma ISO 19136 Annex E encoding rules for Application Schema.
Parte 3: OWL y RDF para la definición de enlaces entre los objetos que implementan el modelo HY_Features, para basarse en la Norma ISO 19150 encoding rules (cuando se aceptado como estándar).
El documento fundacional está disponible en el siguiente enlace https://portal.opengeospatial.org/files/64089
Los comentarios se deben enviar a charter-requests@opengeospatialNOspamPlease.org. El plazo para participar está abierto hasta el próximo 23 de julio.

22/06/15 SensorThings API standard for Internet of Things disponible para comentarios
OGC ha publicado la primera parte del estándar OGC Sensor Things API Standard. Sensor Things API es una especificación de interfaz libre y no propietario que simplifica y acelera el desarrollo de aplicaciones Web basada en el Internet de las Cosas (Internet of Things,IoT).
Desarrolladores de aplicaciones pueden usar este estándar para diversos recursos y crear aplicaciones innovadoras sin la necesidad de integrar protocolos heterogéneos de diferentes aparatos, gateways y servicios. Fabricantes de sistemas pueden integrar OGC SensorThings API en sus plataformas de hardware y software de modo que diferentes plataformas puedan conectarse con otros servidores que implementan el estándar.
Este nuevo estándar está diseñado específicamente por recursos limitados de dispositivos IoT y comunidad de desarrolladores web. El estándar candidato se basa en los principios REST, usa una eficiente codificación JSON y convenciones OASIS OData URL.
Ha sido diseñado para ser compatible con un conjunto de estándares probados y extensamente adoptados como estándares abiertos, como los protocolos Web y OGC Sensor Web Enablement (SWE) standards, incluyendo el estándar SO/OGC Observation and Measurement Encoding Standard. De este modo, Sensor Things API es extensible y puede ser aplicado en casos de uso simples y complejos.
SensorThings API proporciona dos funcionalidades principales, cada uno manejado por un perfil: Sensing Profile y Tasking Profile. El Sensing Profile, es la Parte 1, proporciona el modo de manejar y recuperar las observaciones y metadatos de distintos sistemas de sensores, similar al estándar OGC Sensor Observation Service (SOS) Interface . El Tasking Profile, será definido en documento separado como Parte 2 del estándar, proporcionará funciones similares a OGC Sensor Planning Service (SPS) Interface Standard.
La documentación del estándar se encuentra disponible en http://www.opengeospatial.org/standards/requests/134. El plazo para enviar comentarios está abierto hasta el próximo 18 de julio.
08/05/15 Llamada a la participación en el experimento de Interoperabilidad Soil Data
OGC ha anunciado una Llamada a la participación en experimento de interoperabilidad sobre suelos, Soil Data Interoperabilty Experiment (Soil IE).
Este año ha sido declarado Año Internacional de los Suelos. Es necesario comprender más la distribución de suelo, función y estado para apoyar la ciencia y el desarrollo de política, mejorar la productividad agrícola de manera sostenible y abordar otras cuestiones de alcance mundial como la disminución de diversidad biológica y el cambio climático. Este conocimiento necesita ser sustentado por una evolución cualitativa de suelos para que pueda ser organizada, agregada y accesible.
Técnicas de modelización y mapas de suelo son un gran valor para la toma de decisiones que a menudo dependen de datos dispersos, incongruentes y difícil-a-acceso. Claramente sistemas de información modernos, armonizados e interoperables están obligados a integrar estos datos en un conjunto coherente de información del suelo. Diversas iniciativas han empezado a trabajar en el problema de la integración de datos del suelo mediante la definición de modelos de información del suelo. Por ejemplo: the European INSPIRE, e-SOTER and ISO 28258 (SoilML); the international GlobalSoilMap.net consortium, and the Australian and New Zealand ANZSoilML proyectos. UN FAO Global Soil Partnership reconoce el valor de los proyectos, particularmente Pillars 4 (Global Soil Information System) and 5 (Harmonization). Sin embargo, todavía tenemos una situación donde se debe conciliar múltiples sistemas que tienen los mismos objetivos.
Soil IE pretende permitir la integración de datos para automatizarlo tanto como sea posible. Para ello, International Union of Soil Sciences (IUSS) Working Group on Soil Information Standards (WGSIS) está trabajando para consolidar estos modelos de información y conciliarlas en una sola lengua para el intercambio de información sobre suelos globalmente coherente. WGSIS no espera que los participantes puedan cambiar sus sistemas de información; por el contrario, el objetivo es desarrollar un modelo de datos común de suelos para que cada sistema pueda comunicarse con los demás sistemas de información del suelo.
WGIS propone validar los modelos de información realizando un experimento de interoperabilidad. El experimento de interoperabilidad refinará y probara el SoilML2, consolidando los estándares existentes de suelo testeándolos (a través de implementaciones de trabajo) contra un conjunto de casos de uso acordados para el intercambio y análisis de datos del suelo. Pare ello, WGIS necesita apoyo global y aceptación, construyendo una comunidad globalmente representativa que proporciona tanto dominio y conocimientos técnicos para este proyecto.
WGSIS necesita un núcleo de miembros OGC para apoyar e iniciar este trabajo, se puede participar como observer siendo no miembros. Los iniciadores solicitan a las organizaciones interesadas indicar su apoyo y compromiso de participar en este experimento de interoperabilidad. Los posibles participantes están invitados a asistir a la próximo Comité Técnico en Boulder, Colorado. Los participantes en el Soil IE, probaran las implementaciones y proporcionar comentarios sobre el modelo de intercambio de datos y la documentación de pruebas resultantes.
Los iniciadores del experimento son:
CSIRO (Commonwealth Scientific and Industrial Research Organisation), Australia
Landcare Research New Zealand Ltd.
ISRIC - World Soil Information, Netherlands
Toda la información para participar, programa y la reunión inicial del proyecto está disponible en https://portal.opengeospatial.org/files/?artifact_id=63190&version=1
24/04/15 Llamada a la participación del Major Sensor IoT interoperability pilot project
OGC ha anunciado una Llamada a la participación (RFQ/CFP) en el proyecto OGC Incident Management Information Sharing Internet of Things Pilot Project (IMIS IoT Pilot). Los participantes realizarán prototipos para mostrar nuevos enfoques basados en estándares a un conjunto de retos que impiden el empleo eficaz de gran cantidad de sensores para hacer frente a situaciones de emergencia y catástrofes.
Los proyectos piloto OGC prueban estándares en aplicaciones reales usando productos basados en estándares comerciales Standards Based Commercial Off-The-Shelf (SCOTS) que implementan estándares OGC y otros.
Los patrocinadores han documentado los requisitos y objetivos para este proyecto. Las organizaciones seleccionadas a participar desarrollarán soluciones basadas en los casos de usos de los patrocinadores, requisitos y escenarios, los cuales son descritos en la (RFQ/CFP) http://www.opengeospatial.org/standards/requests/133. 21/01/15 OGC ha adoptado IndoorGML como estándar de codificación
OGC ha anunciado la aprobación de IndoorGML como estándar de codificación. Este estándar especifica un modelo abstracto de datos y esquema XML para información espacial de interiores. El requerimiento impulsor para IndoorGML es la navegación.
IndoorGML proporciona una descripción tecnológica neutra de los espacios navegables de un edificio que pueden ser utilizados por cualquier desarrollador de aplicaciones y proveedores de plataformas ya que IndoorGML es un esquema aplicación del estándar Geography Markup Language (GML), permite una fácil integración de sistemas de navegación de interiores y exteriores. Los desarrolladores de IndoorGML anticipan que esto beneficiará a respuestas de emergencia, compradores, visitantes de aeropuertos y muchos otros ámbitos.
En el sector ya se está usando estándares de modelado 3D como el estándar City Geography Markup Language (CityGML) Encoding Standard , OGC KML y buildingSMARTs Industry Foundation Classes (IFC) . Estos abordan superficies exteriores y tratan con espacios interiores de edificios desde un punto de vista geométrico, cartográfico y semántico. IndoorGML se centra específicamente en modelación de espacios interiores desde un punto de vista de navegación.
En lugar de centrarse en componentes de construcción como tejados, paredes, IndoorGML proporciona un marco que representa espacios cerrados como celdas. La conexión entre espacios y restricciones de movimiento entre espacios (como puertas) son representados como relaciones entre celdas. El estándar define la siguiente información sobre el espacio interior:
Ámbito de navegación y obstáculos
Subdivisiones del espacio y tipo de conectividad entre espacios
Propiedades geométricas y semánticas de los espacios y conectividad
Redes de navegación (lógica y métrica) y sus relaciones
IndoorGML proporciona un estándar abierto que facilita la búsqueda, evaluación, acceso y uso de información para aplicaciones que permiten por ejemplo, servicios basados en localización indoor, análisis de ruta, servicios de geotiquetado (geotagging), gestión de recursos de interiores y control de emergencias. Requisitos para otros tipos de aplicaciones, incluyendo gestión de instalaciones interiores, mapas robot y posicionamiento interior pueden abordarse en la próxima versión de IndoorGML. Además puede ser utilizado para integrar información de muchas tecnologías diferentes de localización de interiores. 13/04/13 Extensión Coordinates Reference System (CRS) del estándar Web Coverage Service (WCS) disponible para comentarios

Web Coverage Service define una interfaz estándar que permite la obtención de datos geoespaciales en forma de coberturas multidimensionales y espacio-temporales, como los datos que se pueden obtener de un sensor de información, imágenes de satélite, series temporales, datos de simulación, nubes de puntos y mallas.
